### ⏸ MCP-WRITING-3 — deferred (by design)

- **Reason for defer:** not a code bug. Could be silenced at source by either (a) making `action` default to `"stats"` or (b) filtering `ValidationError` in Sentry's inbound-data rules. Recommendation: configure a Sentry inbound filter for `pydantic_core._pydantic_core.ValidationError` on MCP tool-arg validation to cut noise.
- **Sentry action:** leave unresolved. No PR.

## Patterns across issues

1. **Deploy-config drift on mcp-writing-library.** MCP-WRITING-A came from adding new file tree (`data/`) without updating `Dockerfile`. Worth adding a regression guard — e.g., a GitHub Action that does `docker build . && docker run --rm <img> python -c "from src.tools.pattern_store import load_values; load_values('para_limits')"`. A smoke test in CI would have caught this pre-deploy.
2. **Qdrant payload-index discipline.** MCP-WRITING-9 is the second indexing-related fix script in this repo (the first created `name`/`channel`/`client_id` indexes). Consider codifying "every filter field gets a payload index" at collection-creation time in `collections.py` → `setup_user_collections`, so a migration script isn't needed for future fields.
3. **Sentry noise from expected-failure MCP validation.** MCP-WRITING-3 has accumulated 8 events in 17 days for what is a protocol-level caller error. An inbound Sentry filter for `mcp.server.fastmcp.exceptions.ToolError` chained from `pydantic.ValidationError` would cut this cleanly.
4. **Triple-counted auth failure.** MCP-AUTH-SERVER-9, -G, -H are three Sentry issues for one logical event (401 → wrapped RuntimeError → logger.error). Consider either (a) not `logger.error`-ing after already `raise`-ing, or (b) Sentry issue-merge rules to collapse these. One `capture_exception` per failure is enough.
