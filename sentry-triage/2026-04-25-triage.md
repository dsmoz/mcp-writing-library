## §2 — Per-issue notes

### 🔀 MCP-WRITING-A — PR awaiting review

`score_writing_patterns` calls `_para_limits()` → `load_values("para_limits")`
→ reads `data/patterns/para_limits.json`. The file lives in the repo
(commit [`43b735f0`](https://github.com/dsmoz/mcp-writing-library/commit/43b735f0))
but the **Dockerfile only copies `src/`, `vendor/`, `main.py`** — the entire
`data/` directory is missing from the production image. So `_read_json`
returns `None` and `load_values` raises `FileNotFoundError("Unknown pattern
file: para_limits")` for every detector that loads from the pattern store
(burstiness, hedging, paragraph length, etc.).

PR [#6](https://github.com/dsmoz/mcp-writing-library/pull/6) (opened by the
2026-04-23 triage run) adds `COPY data/ ./data/` to the Dockerfile. One-line
fix, but it's a deploy-config change so no auto-merge.

### 🔀 MCP-SCHOLAR-Z — PR awaiting review

Caller passes `{"limit": "5", …}` to `search_external_sources`; the deployed
release `1e76412d` (~7 days old) has a handler signature lacking `limit`,
hence `TypeError: unexpected keyword argument 'limit'`. The handler on `main`
already accepts `limit`/`limit_per_source`/`max_results` (see
`src/mcp_tools/direct.py`), but the deployed image is older than the fix.

PR [#2](https://github.com/dsmoz/mcp-scholar/pull/2) is open with the
explicit fix. Just needs review + redeploy.

### ⏸ MCP-SCHOLAR-3G — user `run` script, not a server bug

Stacktrace shows `tool: run`, `<frozen os>`, `<string>`, line 3 — the error
happens inside user-supplied Python code passed to the `run` MCP tool
(`code_length: 4104`). The user's script does
`os.environ['ZOTERO_USER_ID']` and that key is not set in the runtime
exec_globals. A single occurrence; treat as user error. (Optional follow-up:
expose a friendly KeyError → `ToolError("ZOTERO_USER_ID not configured")`
shim in the `run` wrapper, but not actionable here.)

### ⏸ MCP-SCHOLAR-3P / 3N / 3M / 3K — Supabase Session-Mode pool exhaustion

```
FATAL: MaxClientsInSessionMode: max clients reached - in Session mode max
clients are limited to pool_size
```

Four issues, same root cause. PgBouncer/Supabase pooler is in **Session
mode** with a fixed `pool_size`, and the worker fleet exhausts it under
load. Fixes are infrastructure, not code:

1. Switch the pooler to **Transaction mode** (preferred — Supabase
   recommends this for short-lived ORM sessions).
2. Or increase `pool_size` on the Supabase project.
3. Or reduce concurrent worker count.

Hard stop per routine: schema/deps/CI/CD-adjacent. Recommend operator
intervention.

### ⏸ MCP-SCHOLAR-3H / 3J — LLM backend cascade

`MCP-SCHOLAR-3J` is a generic `APIConnectionError` (47 events in 6 h).
`MCP-SCHOLAR-3H` is "All LLM backends failed for entity extraction" (also
47 events in 6 h, identical cadence). 3H is the cascading fallback failure
caused by 3J — when every configured LLM provider is unreachable, the
fallback chain exhausts and entity extraction surfaces a hard failure.

Likely upstream provider outage or container-level egress problem on
Railway. Code already has fallback logic (3H proves the chain ran). No
minimal code fix possible from here — this needs the operator to:

1. Check provider status pages for the configured LLM backends
2. Check Railway egress / network for the `e994333f2703` container
3. Once root cause is known, decide whether to add provider-rotation,
   exponential backoff, or a circuit breaker

## §3 — Cross-issue patterns

1. **Deploy-config drift drives 70% of today's surfaceable bugs.** Both
   actionable issues (`MCP-SCHOLAR-Z`, `MCP-WRITING-A`) are deploy-time
   problems where the code on `main` is correct but the running container
   is stale or missing assets. Recommend wiring Railway auto-deploy on
   merge to `main`, plus a smoke test that validates `data/patterns/*.json`
   is reachable and `tools/list` matches expected signatures.

2. **Old MCP App / `ui://` flows leave orphan errors.** PR #20 cleanly
   pivoted `mcp-writing-library` to HTML-artifact rendering, which made
   the old `ui://review-session/panel` resource and the previous
   `start_review_session` schema obsolete simultaneously. Both
   `MCP-WRITING-3` and `MCP-WRITING-B` are now stale. Worth resolving
   them in Sentry to clear the dashboard once the post-PR-20 deploy is
   confirmed stable.

3. **Supabase pool exhaustion is escalating.** Four separate Sentry issues
   today all bottom out at `MaxClientsInSessionMode`. The pooler config
   needs a one-off fix (Transaction mode) or the issues will keep
   recurring as the worker fleet grows.

4. **The `run` tool is a long tail of low-signal noise.** Today's
   `MCP-SCHOLAR-3G` is the latest in a pattern of user-script errors
   bubbling up as Sentry issues. Consider wrapping `run` execution to
   only forward errors that originate in our code, not in the
   exec'd `<string>` frame.
